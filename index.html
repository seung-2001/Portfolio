Troubleshooting

문제 해결 로그

코드

🔴 문제
지도 검색 페이지에서 키워드 입력 후 Enter로 검색하면, 한 번 누를 때는 아무 반응이 없고, 두 번째 Enter에서야 검색이 됐습니다. 반면 검색 아이콘을 클릭하면 한 번에 잘 동작했습니다.

🔍 원인
fetchReviews 함수가 useCallback으로 정의되어 있고, 내부에서 loading과 hasNext 상태로 "지금 호출해도 되는지"를 검사하고 있었습니다. Enter를 누르면 setCursor(null), setHasNext(true)를 먼저 호출한 뒤 곧바로 fetchReviews(false, null)을 호출하는데, React의 setState는 비동기라서 그 시점에는 아직 hasNext가 이전 값(false)인 상태였습니다.

✅ 해결
검색 버튼/Enter로 "지금 당장 검색"을 할 때는 상태 값에 의존하지 않고 무조건 실행되도록, fetchReviews에 force 파라미터를 추가했습니다. force === true일 때는 loading, hasNext 조건을 보지 않고 바로 API를 호출하게 했고, Enter 키는 onKeyDown + preventDefault()로 처리했습니다.

💡 배운점
상태를 바꾼 다음에 바로 그 상태를 믿고 함수를 호출하면 React에서는 아직 예전 값이 남아 있어서 의도대로 안 돌아갈 수 있었습니다. "지금 이번만은 무조건 실행"이 필요할 때는 force 같은 플래그를 사용하는 것이 좋았습니다.

버그

🔴 문제
신고 API에서 문의(INQUIRY)만 보내고 boardNo 없이 요청하면 400 Bad Request, "게시글 번호가 필요합니다"가 나왔고, 문의를 두 번째로 넣으려 하면 "이미 해당 게시글을 신고하셨습니다"로 막히는 현상이 있었습니다.

🔍 원인
유효성 검사에서 모든 신고에 대해 boardNo를 필수로 검사하고 있었고, 중복 신고 체크도 카테고리 구분 없이 "같은 회원 + 같은 boardNo"로만 하고 있었습니다.

✅ 해결
boardNo 필수 검사는 문의(INQUIRY)가 아닐 때만 하도록 수정했습니다. 중복 신고 확인은 "같은 게시글을 같은 사람이 여러 번 신고"하는 경우만 막고, 문의는 막지 않도록 했습니다.

💡 배운점
하나의 API 안에 "문의"처럼 게시글과 무관한 흐름과 "게시글 신고"처럼 특정 리소스에 묶인 흐름이 섞여 있으면, 유효성과 비즈니스 규칙을 카테고리별로 분기해서 적용해야 합니다.

코드

🔴 문제
레스토랑 목록 API에서 각 레스토랑마다 대표 썸네일을 따로 조회하는 방식으로 구현했는데, 레스토랑이 20개면 쿼리가 1+20번 나가서 N+1 문제가 생겼습니다.

🔍 원인
목록 1번 + 레스토랑 하나마다 썸네일 1번씩 조회하다 보니 데이터가 많아질수록 DB 요청 수가 크게 늘어났습니다.

✅ 해결
MyBatis 매퍼에서 레스토랑 한 행을 조회할 때, 스칼라 서브쿼리로 해당 레스토랑의 대표 썸네일 URL을 한 번에 가져오도록 수정했습니다. 목록 조회 1번으로 레스토랑 정보와 썸네일을 같이 가져오게 했습니다.

💡 배운점
목록 API는 한 번의 쿼리로 필요한 데이터를 모으는 게 중요하다는 걸 배웠고, 설계 단계에서 N+1이 생길지 미리 점검하려고 합니다.
